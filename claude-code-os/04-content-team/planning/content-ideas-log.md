## "Can AI Really Build Your Automations? Here's What YouTubers Won't Tell You"
**Source:** Deep dive analysis of Make.com MCP server timeout issues and automation reality (December 19, 2024)
**Date Logged:** December 19, 2024
**Status:** Logged

### Hook
"YouTubers say AI can build automations for you now - no coding needed, just describe what you want. I just spent 6 months building real automations for clients. Here's the truth they're not telling you."

### Content Structure
1. **Opening:** The marketing claim vs. the reality (AI builds everything! vs. debugging at midnight)
2. **Middle:** What AI ACTUALLY does (70% initial build, but the last 30% requires 100% of the expertise)
3. **Framework:** The 6 Skills AI Can't Replace in Automation Building
4. **Close:** AI is a tool that accelerates experts, not a replacement for understanding your business

### Why It Works
- **Pain Point:** Business owners being sold false promises about "AI does it all" while struggling with broken automations
- **Solution:** Understanding what AI CAN do vs. what YOU need to bring
- **Proof:** Real case study - Lombok Property Scraper optimization (615KB → 370KB, 56% credit savings)

### Key Facts & Data Points

**Automation Complexity Reality:**
- Property scraper blueprint: 370-615KB of JSON configuration
- 5 top-level modules + nested Router with 3 routes
- Each route contains 7-12 modules = ~30-40 total modules
- Complex filter logic using IML (Integromat Markup Language)
- Module references like `{{23.hasKuta}}` vs `{{232.hasKuta}}` (different routes, same data)

**The Optimization Example:**
- Initial version: ~555 credits per run (v4.1, v4.2 at 615KB)
- Optimized version: ~240 credits per run (v5-v7 at 370KB)
- **56% credit savings** achieved through manual analysis
- Required: Understanding filter order, recognizing redundant modules (151, 152, 153), proving mathematical equivalence

**What AI Generated:**
- ✅ Initial workflow structure
- ✅ Basic module configurations
- ✅ Filter syntax examples
- ✅ Documentation lookups

**What AI COULDN'T Do (Human Required):**
1. **Business Logic Understanding**
   - Know that "Kuta" location matters for property filtering
   - Understand construction status varies by website
   - Recognize that properties under $300k was the actual goal

2. **Platform Expertise**
   - Discover Make.com has 300-second timeout limit
   - Realize MCP requires full blueprint replacement (no incremental updates)
   - Understand 370KB+ blueprints hit timeout thresholds
   - Know to use local scripts + UI import instead of MCP direct updates

3. **Cost Optimization**
   - Analyze that price normalization ran on 94 items when only 15 passed filters
   - Calculate 56% savings opportunity by moving filters earlier
   - Understand operation credits system and optimize for it

4. **Debugging Complex Logic**
   - Modules 151, 152, 153 (Data Quality Check) weren't being used
   - Different module IDs for same data across routes ({{23.x}} vs {{232.x}} vs {{233.x}})
   - Filter equivalence verification: prove A→B→C = A→C→B mathematically

5. **Edge Case Handling**
   - Construction status terminology differs by website
   - "Off-Plan" vs "Under Construction" vs "land" in URLs
   - Price formats vary (some need normalization, some don't)

6. **Real-World Testing**
   - Export before/after results
   - Verify 15 properties in, 15 properties out
   - Spot-check data integrity

**Technical Limitations Discovered:**
- Make.com MCP server: External (SSE: https://mcp.make.com/sse)
- Timeout processing for blueprints: 11-18 seconds (370KB) to 23-40 seconds (615KB)
- MCP tool timeout expectations: 3-5 seconds
- Result: Intermittent timeouts forcing manual UI import workflow

**The Required Workflow (Reality):**
```
1. Export blueprint via Make.com UI or scenarios_get
2. Edit JSON locally with Python scripts
3. Validate locally using MCP validation tools
4. Import via Make.com UI (bypasses timeout issues)
```

**Why This Workflow Exists:**
- Make.com MCP only supports full blueprint replacement
- No incremental updates (can't "add module" or "update module X")
- Large JSON payloads (370-615KB) exceed timeout thresholds
- Network latency + processing time = consistent failures
- Local scripts + UI import = only reliable method

**6. Maintenance & Evolution**

### The "No-Code" Misnomer

**Traditional Code:**
```javascript
function filterProperties(properties) {
  return properties.filter(p =>
    p.hasKuta &&
    !p.url.includes('-land') &&
    p.price <= 300000
  );
}
```

**Make.com "No-Code":**
```json
{
  "and": true,
  "conditions": [
    {"a": "{{23.hasKuta}}", "b": "{{true}}", "o": "text:equal"},
    {"a": "{{23.url}}", "b": "-land", "o": "text:notcontain:ci"},
    {"a": "{{23.normalizedPrice}}", "b": "300000", "o": "number:less"}
  ]
}
```

**It's STILL coding** - just in JSON instead of JavaScript. You still need to:
- Understand logic operators (AND, OR, NOT)
- Debug when conditions don't work as expected
- Know data types and operator compatibility
- Understand module references and dependencies
- Handle syntax errors and validation

**What AI/No-Code ACTUALLY Provides:**
- ✅ Faster initial development (days vs. weeks)
- ✅ Lower barrier to entry (don't need to be a programmer)
- ✅ Visual debugging (see flow instead of reading code)
- ✅ Easier iteration (modify without redeploying)

**What It Does NOT Provide:**

- ❌ "Set and forget" automation
- ❌ "AI does it all for you"
- ❌ "No technical knowledge required"

### Platform Comparison Bonus Content

**n8n vs Make.com:**
- n8n: Local MCP (npx n8n-mcp) = lower timeout risk
- Make.com: External MCP (SSE: https://mcp.make.com/sse) = higher latency
- n8n: Native merge node (can recombine split flows)
- Make.com: NO native merge (requires workarounds)
- n8n: Self-hostable (€10/month VPS, full control)
- Make.com: Fully managed (easier handoff to clients)

**When to Choose Each:**
- Make.com: Client deliverables, simpler workflows, easier handoff
- n8n: Complex merge logic, heavy processing, cost-sensitive internal projects
